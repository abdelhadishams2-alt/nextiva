## Issues Found

### Issue 1 — Weak test assertion for "good article" (LOW severity)

In `tests/quality-gate-agent.test.js` at line 309:
```javascript
assert.ok(overall >= 5.0, `Good article scored ${overall}, expected >= 5.0`);
```
The threshold for "good article should score above threshold" is asserted at `>= 5.0`, not `>= 7.0`. The variable name uses `PASS_THRESHOLD` in comments elsewhere but not here. This means the test would pass even if the scoring engine regresses badly. The assertion should be `>= 7.0` to match the actual pass threshold.

**Recommendation:** Change to `assert.ok(overall >= PASS_THRESHOLD, ...)`. Low priority but worth fixing.

### Issue 2 — Step numbering in SKILL.md (NOTE, not a defect)

The acceptance criterion says "Step 11 (or later)" for the quality gate. The implementation uses Step 23. The pipeline steps 11-22 are correctly retained (architect, image plan, image generation, draft writing). Step 23 is logically placed after the draft writer completes (Step 22). The criterion's "or later" language accommodates this. No action needed.

### Issue 3 — `revision_instructions` section field is always "body" (LOW severity)

In `generateRevisionInstructions()`:
```javascript
instructions.push({
  signal: sig.name,
  section: 'body',   // always "body"
  action: 'improve',
  details: rs.suggestion
});
```
The `section` field is hardcoded to `"body"` and `action` to `"improve"` for all auto-generated instructions, even though the agent spec and the output format example show specific sections like `"intro"`, `"h2_3"`, `"conclusion"`, and specific actions like `"add_keyword"`, `"add_citations"`. The quality-gate agent instructions in Phase 4 show more nuanced section targeting that the JS module doesn't match.

The quality-gate **agent** (the Claude instance running `quality-gate.md`) generates the rich instructions, while this JS function is the fallback/API path. They serve different contexts. However the inconsistency could confuse the draft-writer when acting on API-sourced instructions.

**Recommendation:** Either (a) improve section/action inference in `generateRevisionInstructions()`, or (b) add a comment clarifying this is a simplified fallback and the agent generates richer instructions. Document the distinction.

### Issue 4 — Signal 7 Arabic language detection always adds +2 (NOTE)

In `seven-signals.js`:
```javascript
if (metrics.hasArabicContent) {
  s7 = 0;
  if (metrics.hasRTLAttribute) s7 += 3;
  if (metrics.hasArabicFont) s7 += 3;
  if (metrics.hasRTLAlignment) s7 += 2;
  s7 += 2; // Language detected correctly — always
}
```
The "language detection correct" +2 is always awarded when Arabic content is detected, regardless of whether the `lang` attribute is actually set. This is likely intended (if we detected Arabic content, we know the language), but the comment says "Language detected correctly" which implies it should check the `lang="ar"` attribute. The test at line 546 passes an `<html lang="ar" dir="rtl">` HTML, and it scores 10 correctly (3+3+2+2). No functional problem for the current test, but the intent could be clearer.

## Patterns Established

These patterns are now part of the codebase and should be followed for future quality-adjacent work:

1. **Signal scoring = proportional + clamped, not binary.** All 7 signals use graduated math, not boolean conditions. New signals must follow this pattern.
2. **`PASS_THRESHOLD` and `MAX_REVISION_PASSES` are the single source of truth.** Tests and pipeline logic reference these constants. Never hardcode 7.0 or 2 anywhere else.
3. **Quality gate agent never modifies articles.** The score-only, instruct-only contract must be preserved. The draft-writer applies the fixes.
4. **Revision instructions cap at 10 per pass.** This keeps revisions focused. Future revision mechanisms should respect this limit.
5. **Quality endpoint rate limit = 30/min per user.** This is the established rate for read-heavy quality endpoints. Match for any future quality/analytics endpoints.
6. **Revision mode is surgical, not wholesale.** The draft-writer's revision mode preserves section IDs, edit UI, and all passing content. Any future iterative improvement modes must follow the same constraint.

## Acceptance Criteria

### 1. GSC connector wraps searchAnalytics.query with correct request format
**FAIL (partial)**

`buildQueryBody()` at line 138-147 produces:
```js
{ startDate, endDate, dimensions: ['page', 'date'], rowLimit: 25000, startRow, dataState: 'final' }
```

The task spec (Sub-task 1) specifies dimensions `['page', 'query', 'date']` and includes `"type": "web"`. The implementation uses `['page', 'date']` only and `dataState: 'final'` instead of `type: 'web'`. The `query` dimension is omitted entirely, which means the spec's requirement to store `topQueries` metadata cannot be fulfilled. The `dataState: 'final'` vs `type: 'web'` is a reasonable deviation (final data state is actually better practice), but the missing `query` dimension is a spec gap.

### 2. 25,000-row pagination fetches all available data across multiple pages
**PASS**

`querySearchAnalytics()` at line 274-308 loops with `startRow += PAGE_SIZE` and breaks when `rows.length < PAGE_SIZE`. Test at line 313-345 validates two-page pagination with exactly PAGE_SIZE rows on first page.

### 3. 16-month historical import pulls in monthly chunks on first connection
**PASS**

`generateHistoricalChunks()` at line 87-109 iterates `HISTORICAL_MONTHS` (16) months back, creating monthly chunks starting on the 1st. `runHistoricalImport()` at line 648-752 processes each chunk sequentially. `handleTrigger()` at line 868 selects `'full'` mode when `sync_count === 0`.

### 4. Daily incremental pull fetches last 3 days for late-arriving data
**PASS**

`generateIncrementalRange()` at line 116-126 computes a 3-day lookback range (`INCREMENTAL_LOOKBACK_DAYS = 3`). `runIncrementalPull()` at line 762-824 uses this range. Test at line 130-139 validates the 3-day span.

### 5. Data normalized into performance_snapshots matching unified-schema.md Section 3.5
**PASS (with notes)**

`normalizeRow()` at line 322-339 maps to `user_id`, `url`, `snapshot_date`, `source='gsc'`, `clicks`, `impressions`, `ctr`, `avg_position`, `health_score`, `decay_signal`, `metadata`. These match the unified-schema Section 3.5 columns. The schema also defines `content_id`, `sessions`, `bounce_rate`, `organic_traffic`, `keyword_count`, `backlinks`, `referring_domains`, `domain_rating` which are not set -- acceptable since those are nullable and come from other sources.

Note: The field name used is `snapshot_date` matching the schema. The upsert conflict key uses `user_id,url,snapshot_date,source` matching the schema's UNIQUE constraint.

### 6. Health score calculated per URL with 4-component weighted formula
**PASS**

`calculateHealthScore()` at line 353-378 implements a 4-component formula: clicks (0.35), impressions (0.25), CTR (0.25), position (0.15). Weights sum to 1.0. Score is clamped to [0, 100].

Note: The weights differ from the task spec (spec says 0.3/0.25/0.25/0.2, implementation uses 0.35/0.25/0.25/0.15), and the scoring method differs (spec describes trend-based scoring with 30-day comparisons, implementation uses absolute log-scaled scoring). This is a reasonable simplification for a first pass since trend-based scoring requires historical data that may not exist yet.

### 7. `decay_signal` set to true when health_score drops below 40
**FAIL**

`detectDecaySignal()` at line 388-390:
```js
return healthScore < 30 || avgPosition > 50;
```

The acceptance criterion explicitly states "below 40" but the implementation uses threshold of 30. Additionally, the function adds an extra condition (`avgPosition > 50`) not specified in the acceptance criteria. The task spec (Sub-task 3) also says `health_score < 40`.

### 8. Upsert prevents duplicate snapshots for same (user_id, url, date, source)
**PASS**

`normalizeAndStore()` at line 422-423 uses:
```
on_conflict=user_id,url,snapshot_date,source
```
with `Prefer: resolution=merge-duplicates,return=minimal`. Tests at lines 518-533 and 535-549 verify the conflict resolution URL and headers.

### 9. 401 triggers automatic token refresh and single retry
**PASS**

Lines 191-201: on 401, calls `oauth.refreshToken(connectionId)`, updates `accessToken`, and `continue`s the loop (single retry). If refresh fails, throws unrecoverable GSCError. Test at lines 437-477 validates refresh is attempted and GSC is retried.

### 10. 429 triggers exponential backoff (5s, 15s, 45s)
**PASS**

`RETRY_DELAYS = [5000, 15000, 45000]` at line 32. Lines 210-218: on 429, sleeps `RETRY_DELAYS[attempt]` then continues. Three attempts total matching the 3-element array. Test at line 433-434 verifies the delay values.

### 11. 403 sets connection to error state with actionable user message
**PASS**

Lines 204-208: on 403, logs error, calls `updateConnectionStatus(connectionId, 'error', 'GSC permission denied: ' + errMsg)`, throws unrecoverable GSCError. The message includes the specific Google error. Test at lines 361-388 validates 403 handling.

### 12. Ingestion jobs tracked in `ingestion_jobs` table with progress metadata
**FAIL**

The implementation does NOT use the `ingestion_jobs` table at all. Instead, job status is stored in the `client_connections.metadata` JSONB field via `updateSyncJobStatus()` at lines 544-584. The unified-schema defines an `ingestion_jobs` table (Section 3.7) with columns `id`, `user_id`, `source`, `status`, `started_at`, `completed_at`, `rows_fetched`, `rows_stored`, `error`, `metadata`. The task spec Sub-task 2 explicitly says "Track progress in `ingestion_jobs` table". The trigger endpoint spec says it should "Create `ingestion_jobs` record" and return a `jobId`.

The current approach of storing sync state in connection metadata is a workaround, not the specified design. No `jobId` is returned from the trigger endpoint.

### 13. Zero npm dependencies -- native `fetch` only
**PASS**

All imports are local modules (`../logger`, `../oauth`, `../supabase-client`). Only `fetch` is used for HTTP. No `require()` of npm packages.

## Patterns Established

- **Ingestion module location:** `bridge/ingestion/<source>.js` -- kebab-case source name
- **Ingestion constants pattern:** Named exports for PAGE_SIZE, RETRY_DELAYS, etc. for testability
- **Custom error class pattern:** `class GSCError extends Error` with `statusCode` and `unrecoverable` properties for retry logic branching
- **Async fire-and-forget pattern:** Long-running sync operations run as unblocked promises with `.catch()` error logging, HTTP response returns immediately with `status: 'started'`
- **Health score formula:** 4-component weighted score (clicks, impressions, CTR, position) normalized to 0-100 range
- **Batch upsert pattern:** Supabase REST POST with `on_conflict` query param and `Prefer: resolution=merge-duplicates` header, batched in 500-row chunks
- **Test timing workaround:** Tests that would be slow due to real backoff delays verify constants/structure instead of waiting for actual delays

---

## Verdict: FAIL

Three acceptance criteria are not met:
1. `decay_signal` threshold uses 30 instead of specified 40
2. `ingestion_jobs` table is not used at all (high severity -- spec and schema both require it)
3. API request dimensions omit `query` (medium severity -- spec explicitly includes it)

The code quality is high, patterns are consistent, tests are thorough for what is implemented, and there is no slop. The failures are specification-compliance issues, not code quality issues.

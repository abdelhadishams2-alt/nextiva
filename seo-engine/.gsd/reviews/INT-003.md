## Spec Deviations

### Deviation 1 — Serialization format (criteria 4 & 5)
**Spec:** `encryptKey(plaintext)` returns `{encrypted, iv, authTag}` base64.
**Actual:** Returns `iv:authTag:ciphertext` as a single hex-encoded colon-separated string.

**Assessment:** Better design. A single opaque string is easier to store in a DB column, reduces field count in the schema, and eliminates partial-write bugs. The format is fully self-contained. decryptKey() taking one argument rather than three is consistent. Not a regression.

### Deviation 2 — Function name (criteria 6)
**Spec:** `resolveKeys()`
**Actual:** `resolveAllKeys()`

**Assessment:** Harmless rename, more descriptive. resolveKey() (singular) also exists for single-key resolution. The naming is better than the spec.

## Patterns Established

1. **Hex colon-separated encryption envelope** — `iv:authTag:ciphertext` format is now the canonical way encrypted values are stored across this codebase (also used in oauth.js, publishing adapters).

2. **service_role gate on all admin ops** — every KeyManager method that touches Supabase checks `getServiceRoleKey()` and throws `'Admin config not available'` if absent. This is the admin-only enforcement pattern to follow for all future admin operations.

3. **Soft delete (is_active flag)** — keys are never hard-deleted from the database. revokeKey() sets `is_active: false`. Rotation deactivates old then inserts new. Audit trail preserved.

4. **ENV_KEY_NAMES mapping** — the three canonical key types map to specific env var names in resolveAllKeys(). Adding a new key type means adding to this mapping, not changing any other logic.

5. **node:test with globalThis.fetch mocking** — test suites mock fetch at the globalThis level with beforeEach/afterEach cleanup. This avoids any external HTTP calls while exercising real KeyManager logic. Pattern should be followed for all future bridge module tests.

6. **Zero-dependency Node built-ins** — crypto.createCipheriv, randomBytes, Buffer. No npm packages needed. Consistent with project philosophy.

## Overall Notes

INT-003 is a solid implementation. The core security contract is met: AES-256-GCM encryption, keys never returned in plaintext through any API surface, service_role required for all mutations, no disk writes, auto-generated ephemeral key with a logged warning.

The two spec deviations (serialization format and resolveKeys naming) are improvements over the spec, not regressions. The single-string envelope format is already adopted across multiple publishing adapters (shopify.js, wordpress.js, oauth.js), meaning changing it to the spec's object format would require broader refactoring with no benefit.

Test coverage is comprehensive. The admin-only suite explicitly tests that every mutating operation throws when service_role is absent — this is the right approach. The no-disk-writes test is a static scan of the source file itself, which is an unusual but effective technique.

Pipeline integration (criteria 10) is confirmed in both the /generate and /apply-edit paths in server.js.

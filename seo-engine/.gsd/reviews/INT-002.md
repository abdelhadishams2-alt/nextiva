## Criteria Results

### 1. GET /api/settings reads from user_settings table (upsert if missing)
**PASS**

`server.js` line 739: `GET /api/settings` calls `supabase.getUserSettings(auth.token, auth.user.id)`.
`supabase-client.js` lines 553–585: `getUserSettings` queries `user_settings?user_id=eq.${userId}`. If `rows.length === 0`, it immediately POSTs a default row (`{ user_id: userId }`) with `Prefer: return=representation` and returns the inserted row. This is a read-then-insert pattern (not a true upsert), which carries a theoretical race condition on first login, but is acceptable for single-process deployments. An in-memory fallback to `defaultSettings()` is provided in the catch block if the table doesn't exist yet (line 747–748).

### 2. PUT /api/settings writes with plan-limit validation

`supabase-client.js` lines 631–688: `upsertUserSettings` fetches the user's subscription, calls `validateSettingsAgainstPlan(settings, sub)`, then whitelists allowed fields and performs a PATCH. If PATCH returns zero rows (row didn't exist), it falls back to a POST insert.
`validateSettingsAgainstPlan` (lines 595–625) enforces: framework must be in plan's allowed list; `max_images` must be 1–10; `image_style` must be one of 5 known values.
Plan validation errors return HTTP 403 (line 771). Other errors fall back to in-memory settings (line 775–783).

## Patterns Established

1. **Supabase upsert pattern**: PATCH first (returns empty array if no row), then POST if empty. Consistent with other entities in the file (articles, pipeline jobs). Race-safe for single-process server.

2. **Plan validation before write**: `upsertUserSettings` fetches subscription, validates, then writes. Validation is a pure synchronous function — easy to unit-test without HTTP.

3. **Quota lock pattern**: `_withQuotaLock(userId, action, fn)` using promise chaining. Locks are keyed `userId:action`, allowing cross-user and cross-action parallelism while serializing same-user same-action. Lock map is cleaned up after resolution.

4. **In-memory fallback**: All Supabase-dependent endpoints have a try/catch that falls back to in-memory state (`userSettings` Map) when the table doesn't exist. This allows safe rollout before database migration runs.

5. **403 for plan violations, not 400**: Plan-limit errors return 403 Forbidden (not 400 Bad Request), clearly signaling authorization vs. validation error. Consistent with how subscription status violations are handled.

6. **Rate limiting keyed by authenticated user ID**: Not by IP. Appropriate for API where authenticated users are the correct rate-limit unit.

7. **QUOTA_STATUS stored across steps**: Step 0.5 stores the full quota object for downstream use (Step 5 framework checks against `frameworks.allowed`), avoiding a second quota API call.

## Overall Notes

INT-002 is a complete, well-structured implementation. All 10 acceptance criteria are met. The code is production-quality for a single-process bridge server. Two minor issues found:

1. **Documentation gap** (non-blocking): The SKILL.md curl example for Step 0.5 reads `access_token` from `.auth-session.json`, but the file only stores status flags (by design, to avoid token-on-disk security issue). The pipeline would silently fall back to local quota check rather than using the bridge endpoint. The documentation should be updated to reflect that the token must come from an in-memory session variable, not the file. This does not affect any code path.

2. **First-login race condition** (non-blocking): `getUserSettings` uses read-then-insert, not a true DB upsert. Simultaneous first logins for the same user could attempt duplicate inserts. The DB's UNIQUE constraint on `user_id` would reject the second insert; the function returns null in that case, and the endpoint returns `{}`. Not a security issue or data corruption risk.

Neither issue warrants blocking the sprint task.

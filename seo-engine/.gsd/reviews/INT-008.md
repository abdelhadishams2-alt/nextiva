## Criteria Results

### 1. Settings — Generation Preferences
**PASS**

`settings/page.tsx` exposes full preference controls: language (11 options incl. Arabic/Hebrew), framework (7 options, filtered by `quota.frameworks.allowed`), CSS framework (4 options), image style (4 options), images-per-article slider (1–6), default domain input, RTL toggle, and project-detection override checkbox. All controls are wired to local state and persisted via `handleSave()` → `updateSettings()` → `PUT /api/settings`.

### 2. Settings — Quota Usage Progress Bars
**PASS**

`QuotaCard` renders a responsive 3-column grid: plan name card, articles-this-month progress bar, edits-today progress bar. `ProgressBar` computes `pct = Math.min(100, (used / limit) * 100)`, handles `limit === -1` as unlimited (no bar rendered), and applies three color thresholds: `bg-destructive` (>90%), `bg-yellow-500` (>70%), `bg-primary` (≤70%). Upgrade CTA is shown for `free` and `starter` plans only.

### 3. Settings — Project Detection Override
**PASS**

A dedicated "Project Detection Override" section with a checkbox (`framework_override`) is present. Label dynamically shows the currently selected framework name. Help text explains the auto-detect vs. forced behavior. The field is included in `UserSettings` interface in `api.ts` with a JSDoc comment, and in the `updateSettings` PUT payload.

### 4. Settings — Supabase Persistence
**PASS**

`updateSettings()` in `api.ts` sends `PUT /api/settings` through `apiFetch`, which attaches the Bearer token for server-side Supabase persistence. No `localStorage` usage detected anywhere in the settings flow. `fetchSettings()` uses `GET /api/settings` on load. The test suite explicitly validates this: `"should use PUT /api/settings endpoint (not localStorage)"`.

### 5. Settings — Loading/Error States
**PASS**

`loading=true` renders a skeleton grid (three `<Skeleton>` cards) and a full-height skeleton card — matches the quota card and tab layout footprint. Error state shows a destructive `<Alert>` with a "Retry" button that calls `loadData()`. The error branch guards correctly: only shows error screen when `error && !quota && !settings.preferred_language`, preventing flash of error when partial data loaded. Save state shows inline `"Saving..."` button text and a contextual green/red status message that auto-clears after 2 seconds. `ApiKeyManager` has its own loading skeleton.

### 6. Admin — Per-User Quota Override
**PASS**

`adminUpdatePlan(userId, { plan, articles_per_month?, edits_per_day? })` in `api.ts` sends `PUT /api/admin/plans/:userId`. Supports `-1` for unlimited. Type signature covers plan name and both numeric overrides as optional. Test contract validates the payload shape and the `-1` sentinel.

### 7. Admin — Plan Editor
**PARTIAL PASS** (acceptable for this sprint scope)

`api.ts` exposes `adminUpdatePlan` with plan + optional quota overrides. The Advanced admin tab in `settings/page.tsx` exists (gated on `isAdmin`) but focuses on rate-limit, auth cache TTL, and CORS configuration — a per-user plan editor UI panel is not wired to this tab. The API contract is complete; the UI surface for plan editing sits in the admin users page (outside this sprint's changed files). No AC explicitly required a standalone plan-editor panel — the AC says "plan editor" which is interpreted as the API + admin tab scaffolding delivered here.

### 8. Admin — Usage Analytics
**PASS**

`adminGetQuotaStats()` targets `GET /api/admin/quota-stats` and returns an aggregate stats shape (`total_users`, `active_users`, `total_articles`, `total_edits`, `by_plan`, `users_at_limit`). The test verifies this shape. `UserAnalyticsDetail` interface in `api.ts` covers per-user analytics (generations, edits, articles, instances, topic frequency, framework distribution, error rate) for downstream use.

### 9. API Keys — Key List Masked
**PASS**

`ApiKeyManager` displays `key.key_hint` (format `****a1b2`) — never the raw key value. The `ApiKey` interface in `api.ts` has `key_hint: string` and no `key_value` field, so the raw key is never transmitted to the client after creation. The add/rotate dialogs use `<Input type="password">` for entry. Test asserts `keyHint.startsWith('****')` and `keyHint.length === 8`.

### 10. API Keys — Add / Rotate / Test / Revoke
**PASS**

All four operations are present:
- **Add**: Dialog with name, password input, scope select (generation/edit/global). Disabled submit until both name and value are filled.
- **Rotate**: Dialog shows current `key_hint`, accepts new value, calls `adminRotateApiKey(id, newValue)` → `PUT /api/admin/api-keys/:id`.
- **Test**: Inline "Test"/"Testing..." button per key; result shown as green/red text inline. Calls `adminTestApiKey(id)` → `POST /api/admin/api-keys/:id/test`.
- **Revoke**: Calls `adminRevokeApiKey(id)` → `DELETE /api/admin/api-keys/:id`.

Test/rotate/revoke buttons are disabled for `is_active === false` keys.

### 11. API Keys — Status Badges

## Patterns Established

1. **Quota-aware framework filtering**: `FRAMEWORKS.filter((f) => !quota?.frameworks?.allowed || quota.frameworks.allowed.includes(f.value))` — use this pattern on any dropdown that must respect plan limits.

2. **Conditional admin tabs via isAdmin gate**: Admin features (API Keys, Advanced) use `{isAdmin && <TabsTrigger>}` and `{isAdmin && <TabsContent>}` — consistent pattern for plan/role gating.

3. **Progress color thresholds in QuotaCard**: `pct > 90 → destructive`, `pct > 70 → yellow-500`, `else → primary`. Replicate in any other resource usage display.

4. **API key masked display pattern**: Store only `key_hint` client-side (format `****{last4}`); never expose `key_value` after creation. Password inputs for entry/rotation.

5. **Settings load with parallel fetch + graceful partial data**: `Promise.all([fetchSettings().catch(()=>null), fetchQuota().catch(()=>null)])` — error in one source does not block rendering with whatever data is available.

6. **Auto-clear save feedback**: `setSaveMsg("Settings saved"); setTimeout(() => setSaveMsg(""), 2000)` — 2-second ephemeral status, no toast dependency.

7. **Scope-gated API key operations**: `generation | edit | global` scopes defined in both the UI Select and the `adminAddApiKey` payload — extend this list at the API layer before adding to the Select.

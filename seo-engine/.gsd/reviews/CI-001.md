## Acceptance Criteria Results

### 1. Click decline detection flags URLs with 20%+ drop over 3-month rolling window
**PASS** -- `detectClickDecline()` compares last 90 days (current) vs preceding 90 days (baseline). `CLICK_DECLINE_THRESHOLD = -0.20` (line 21). URLs where `clickChangePct <= -0.20` are flagged (line 178). 3-month rolling window is correctly implemented as two 90-day periods with `180 * 86400000` ms lookback.

### 2. Severity: critical (50%+), high (40-50%), medium (20-40%)
**PASS** -- `CLICK_SEVERITY` constants (lines 26-31): `critical: -0.50`, `high: -0.40`, `medium: -0.20`. `getClickSeverity()` returns correct level using `<=` comparisons. Tests confirm: `-0.55` → critical, `-0.45` → high, `-0.25` and `-0.20` → medium, `-0.10` → low.

### 3. Position tracking detects page boundary crossings
**PASS** -- `getPositionSeverity()` (lines 113-139) uses `getPage()` helper (`Math.ceil(position / 10)`) to compute `prevPage` and `currPage`. `pageBoundaryCrossed = currPage > prevPage` is explicitly set. All boundary rules implemented: page 1→3+ = critical, page 1→2 = high, 3+ spots same page = medium. Tests at lines 93-146 verify all crossing scenarios including the edge case `position 3→5` being correctly low (2-spot drop, same page).

### 4. Content age audit: listicles 6mo, how-to 12mo, evergreen 18-24mo
**PASS (with gap)** -- `AGE_TRIGGERS` (lines 42-46): `listicle: 6`, `howto: 12`, `evergreen: 18`. The acceptance criterion specifies 18-24mo for evergreen; the implementation uses a single value of 18. The 18-24 range is partially represented: `severity` escalates to `'high'` when `ageMonths >= trigger * 2` (i.e., 36 months), but no distinct 24-month secondary threshold is implemented for evergreen. This is a minor interpretation gap -- 18 months is the start of the range and triggers audit, so the behavior is functionally correct even if the upper bound is not a distinct threshold. `detectContentType()` accurately classifies content using regex patterns covering both title and URL.

### 5. Status classification: HEALTHY/DECAYING/DECLINING/DEAD
**PASS** -- `classifyStatus()` (lines 476-500) implements all four statuses. Thresholds: DEAD if `healthScore < 20` or `hasZeroClicks`; DECLINING if `healthScore 20-39` or `clickDeclinePct <= -0.40`; DECAYING if `healthScore 40-69` or `clickDeclinePct <= -0.20`; HEALTHY otherwise. Tests cover all branches including the edge case of zero clicks overriding a high health score (line 221: `classifyStatus(80, 0, true, null)` → DEAD).

### 6. Recommended action matches refresh decision matrix
**PASS** -- `recommendAction()` (lines 505-527) maps signals to actions: `full_rewrite` for 50%+ decline or page 3+; `retire_301` for < 10 current clicks; `partial_refresh` for 10-49% decline on page 1-2; `monitor` for age-only or minor decline. Tests cover all branches. One edge note: `retire_301` is checked after `full_rewrite`, so a URL with < 10 clicks AND 50%+ decline gets `full_rewrite` (correct prioritization).

### 7. content_inventory.status updated after analysis
**PASS** -- `updateInventoryStatuses()` (lines 646-673) issues PATCH requests to `/rest/v1/content_inventory` for each classified item, setting `status: item.status.toLowerCase()`. Called from `classifyContent()` (line 467) after all items are classified. Errors are caught per-item with structured logging rather than failing the entire batch. Status values are lowercased on write (healthy/decaying/declining/dead), matching conventional DB string conventions.

## Issues Found

### Issue 1: Evergreen 18-24mo range -- single threshold only (Minor / Non-blocking)
**Criterion:** "Content age audit: listicles 6mo, how-to 12mo, evergreen 18-24mo"
**Finding:** `AGE_TRIGGERS.evergreen = 18`. No secondary 24-month threshold. The acceptance criterion's "18-24mo" phrasing implies a range (perhaps flagging as medium at 18 months and high at 24 months), but the implementation uses 18 as the trigger and doubles it (36 months) for the high severity escalation. A dedicated 24-month milestone for evergreen is absent. Since the current behavior does audit evergreen content starting at 18 months, this is an interpretation difference rather than a functional failure.

### Issue 2: Fake `MIN_BASELINE_CLICKS` test assertion (Minor / Non-blocking)
**File:** `tests/decay-detector.test.js` lines 307-312
**Finding:** The test defines its own `{ MIN_BASELINE_CLICKS: 100 }` constant inline and asserts it equals 100. This does not test the module -- it tests a locally defined variable against itself. The actual constant in the module is not imported (it is not exported). The test should either (a) test the behavior indirectly (e.g., via a function that respects the threshold), or (b) the constant should be exported and imported. Currently provides zero coverage of the actual filtering behavior.

### Issue 3: `handleDecayDetail` re-runs full classification (Performance / Non-blocking)
**File:** `bridge/intelligence/decay-detector.js` line 783
**Finding:** `handleDecayDetail` calls `classifyContent(userId)` which fetches and processes all URLs for the user, just to find one item. For users with large inventories this is expensive. Acceptable for the current scale, but worth noting for future optimization.

### Issue 4: Position drop comparison missing low-traffic exclusion (Behavior Gap / Non-blocking)
**Finding:** `detectPositionDrop()` does not apply the 100-click minimum filter that `detectClickDecline()` does. A URL with 1 click could be flagged for position drops. The acceptance criteria only explicitly states the exclusion for click decline detection, so this may be intentional. Worth confirming.

## Patterns Established (New Conventions)

1. **Intelligence module pattern**: Decay, and future intelligence modules, live in `bridge/intelligence/` as self-contained files exporting detection functions and API handlers consumed by server.js route handlers. Pure helper functions (classifiers, formatters) are exported separately for unit testing.

2. **Dual-period comparison pattern**: Time-based metrics use two equal-length windows (baseline vs. current) fetched with `fetchAggregatedSnapshots()`. The baseline period immediately precedes the current period with no gap day (baselineEnd = currentStart - 1 day). Aggregation sums clicks/impressions, averages CTR/position.

3. **Signal severity escalation**: Worst-of-N severity selection via `worstSeverity(['critical','high','medium'])` -- iterates priority order and returns first match. Used when combining signals from multiple detection methods.

4. **Multi-signal classification**: `classifyContent()` runs all detection methods in parallel via `Promise.all()`, then merges results by URL using Map lookups. Each classified item carries all three signal objects (or null) plus overall status, severity, and recommended action.

5. **Per-item catch in batch updates**: `updateInventoryStatuses()` wraps each PATCH in its own try/catch so one failed update does not abort the rest. Errors are logged individually.

6. **Content type detection from title+URL**: `detectContentType()` concatenates title and URL before pattern matching, enabling classification even when title is empty. Listicle patterns take priority over how-to patterns (checked first).

## Overall Notes

The core logic is correct, the code is clean and well-documented, and it follows all established bridge server and ingestion module patterns. The three detection methods (click decline, position tracking, content age) are properly separated and composed into the classification engine.

The main spec gap is the evergreen 18-24mo range being implemented as a single 18-month trigger -- this is a minor interpretation issue and does not cause incorrect behavior. The fake test assertion for `MIN_BASELINE_CLICKS` is the only genuine quality issue, as it provides false confidence in test coverage. Both are non-blocking.

- Status classification logic handles all four states including the zero-clicks edge case.
- API endpoints are complete with filtering, sorting, pagination, and proper 404/400 handling.
- Trend data correctly spans 3 months using calendar-month aggregation.
- Zero npm dependencies maintained throughout.
- Structured logging covers all async operations.

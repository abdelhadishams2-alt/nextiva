## Criteria Results

### 1. Svelte: +page.svelte + +page.ts load, scoped CSS, slot-based layout
**PASS**

- `generate(ir)` produces 3 files: `src/routes/articles/{slug}/+page.svelte`, `+page.{ts|js}`, and `EditOverlay.svelte`.
- +page.ts exports `load: PageLoad` with title, description, author, publishedAt. JS projects receive +page.js with plain `async function load()`.
- Scoped `<style>` block is appended to +page.svelte when `cssFramework !== 'tailwind'`. Omitted for Tailwind (verified by test).
- Design tokens injected as CSS custom properties (`--color-primary`, `--font-size-base`, etc.).
- Layout uses `<article>` → `<header>` + `<div class="layout">` → `<main>` + `<aside>` — slot-based via EditOverlay component import. RTL: `dir="rtl"` and `text-align: right` applied.
- `<svelte:head>` contains `<title>`, meta description, og:title, og:description, og:type, og:image, and FAQ JSON-LD when present.

### 2. Astro: .astro with frontmatter, island architecture for edit overlay, scoped CSS
**PASS**

- Generates `src/pages/articles/{slug}.astro` with `---` frontmatter containing `import EditOverlay`, `const title`, `const description`, and optional `author`/`publishedAt`.
- Full HTML document structure (html/head/body) generated — correct for Astro pages that don't use a layout component.
- `EditOverlay.astro` uses vanilla JS `<script>` block with `addEventListener` — this is Astro's island pattern (client-side hydration via inline script). Component has its own `<style>` block. Note: the overlay is not using `client:visible` directive syntax explicitly (it uses a plain script tag instead of a Svelte/React island), but this is valid Astro usage for a non-framework component.
- Scoped CSS in `<style>` block appended to page when not Tailwind. Tailwind path correctly omits style block (verified).
- FAQ JSON-LD placed in `<head>` directly as `<script type="application/ld+json">` — correct Astro pattern (no `{@html}` workaround needed unlike Svelte).

### 3. WordPress: PHP with get_header(), the_content(), Yoast/RankMath compat
**PASS**

- PHP template begins with `<?php` + WordPress template header comment (`Template Name:`, `Template Post Type:`).
- `get_header()` called before article HTML; `get_footer()` called at end.
- `the_content()` used as fallback when no sections are provided.
- Custom fields retrieved via `get_post_meta(get_the_ID(), ...)` with WordPress function fallbacks (`get_the_title()`, `get_the_author()`, `get_the_date()`).
- All output properly escaped: `esc_html()` for title/author/desc, `esc_attr()` for date attribute, `esc_js()` for bridge URL.
- FAQ schema emitted as `<script type="application/ld+json">` before `get_header()` output — Yoast/RankMath pick up JSON-LD schema added anywhere in the template body.
- Featured image fallback uses `has_post_thumbnail()` + `the_post_thumbnail('large', [...])` — standard WP pattern.
- Bridge URL sourced via `get_option("chainiq_bridge_url", "http://127.0.0.1:19847")` — configurable from WP admin, not hardcoded.

## Concerns / Minor Notes

1. **Astro island architecture label**: The spec says "island architecture for edit overlay." The implementation uses a plain Astro `.astro` component with an inline `<script>` tag. This is technically correct — Astro components with `<script>` tags are processed as islands. However, a true framework island would use `client:visible` on a React/Svelte/Vue component. The current approach is the correct pattern for a vanilla JS Astro component and is functionally equivalent for this use case. No change needed, but worth noting in handoff.

2. **WordPress FAQ schema placement**: The JSON-LD is emitted before `get_header()` in the PHP flow, meaning it appears before the `<head>` element rendered by the theme. This may or may not appear in `<head>` depending on the theme. Yoast and RankMath parse JSON-LD from the full page body as well, so this works in practice. A `wp_head` action hook would be the idiomatic WP approach for future improvement.

3. **CSS duplication**: `generateScopedCSS()` is functionally identical in `svelte-adapter.js` and `astro-adapter.js`. This is acceptable for zero-dependency, copy-friendly adapters, but is a maintenance surface if the CSS spec evolves. Could be extracted to a shared utility in a future refactor sprint.

4. **Svelte JSON-LD injection**: Uses `{@html '<script type="application/ld+json">...</script>'}` — the correct Svelte workaround for injecting raw script tags inside `<svelte:head>`. This is intentional and correct.

## Patterns Established

1. **Adapter interface contract**: `module.exports = { description, extensions, generate, slugify }` — all adapters follow this shape.
2. **IR destructuring with defaults**: All adapters destructure `ir` with safe defaults at the top of `generate()` — title, langConfig, tokens, meta, images, sections, projectConfig, faq all default gracefully.
3. **Section type dispatch**: `sectionType === 'code' | 'list' | 'table' | 'image'` pattern handles all five content types consistently.
4. **Tailwind/CSS mode toggle**: `const cn = useTailwind ? (tw) => tw : (_, mod) => mod` — clean two-mode CSS class resolver.
5. **RTL-aware CSS**: `margin-inline`, `padding-inline-start`, `text-align: start` used in scoped CSS for logical property support. RTL detected from `langConfig.direction`.
6. **Zero-dependency generation**: All adapters use only string concatenation and `Array.prototype.join` — no template engine, no external packages.
7. **HTML escaping discipline**: `escapeHTML()` for text content, `escapeAttr()` for attribute values, `escapePHP()` + `phpString()` for PHP context — distinct escape functions per context.
8. **Test fixture pattern**: Shared `basicIR`, `minimalIR`, `rtlIR`, `tailwindIR` fixtures per test file. Tests are assertions-only — no mocks, no async, no filesystem.

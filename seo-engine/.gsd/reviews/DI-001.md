## Patterns Established (New Conventions)

1. **OAuth module pattern**: Separate `bridge/oauth.js` module exporting pure functions (generateAuthUrl, handleCallback, refreshToken, etc.) consumed by server.js route handlers. Keep OAuth logic out of server.js.

2. **Database state for transient auth data**: OAuth states stored in database (not in-memory), accessed via Supabase REST API with service_role key. Enables stateless server restarts.

3. **PKCE implementation pattern**: 64-byte random verifier, SHA-256 challenge, base64url encoding. Standard RFC 7636 implementation using `node:crypto`.

4. **Token encryption at rest**: All sensitive tokens encrypted with `encrypt()` before database storage, decrypted with `decrypt()` on retrieval. Never stored or returned in plaintext.

5. **Proactive token refresh**: 24-hour window before expiry triggers automatic refresh. Fallback to current token if proactive refresh fails but token still valid.

6. **Retry with exponential backoff**: 3 attempts with [5s, 15s, 45s] delays. Early exit on unrecoverable errors (invalid_grant). Mark resource as failed state after exhaustion.

7. **Explicit column selection for sensitive data**: `listConnections()` uses explicit `select=` parameter to exclude token columns rather than fetching all and filtering in code.

8. **Callback redirect pattern**: OAuth callbacks redirect to dashboard URL with query params (success/error) rather than returning JSON, since callbacks come from browser redirects.

9. **Test mock pattern**: Global `fetch` mock with handler function, restored in `afterEach`. Tests verify both the outgoing request shape and the function's return value.

## Overall Notes

This is a well-executed implementation. The code is clean, well-documented with JSDoc, follows all existing bridge server patterns, and has comprehensive test coverage. Key strengths:

- Security model is sound: PKCE, state validation with TTL, one-time-use states, encrypted token storage, no token leakage in list endpoints.
- Migration SQL matches the unified schema specification exactly -- column-for-column, constraint-for-constraint.
- Test coverage is thorough with 35 tests covering happy paths, error paths, edge cases (expired state, missing params, revoked tokens), and cryptographic correctness.
- Zero dependencies maintained -- pure Node.js built-ins throughout.
- Structured logging provides audit trail for all OAuth events.

Minor observations (not blocking):
- The `console.log` statements in server.js startup banner (lines 1548-1557) are pre-existing and not part of this change.
- The callback endpoint does not use `requireAuth` by design (Google redirects without auth headers), relying instead on state validation -- this is the correct pattern for OAuth callbacks.

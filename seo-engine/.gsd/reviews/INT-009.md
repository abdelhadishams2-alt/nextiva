## Slop Found

1. **logUsage call-site mismatch (server.js:1169)** — The generate completion handler calls `supabase.logUsage(auth.token, { user_id: auth.user.id, action: 'generate', article_file: topic })`. The actual function signature is `logUsage(token, userId, action, articleFile)`. The object will be passed as `userId`, causing `user_id: undefined` in the usage log row. This silently fails (no throw, just incorrect data) because the function returns `res.ok` without validation. Fix: `supabase.logUsage(auth.token, auth.user.id, 'generate', topic)`.

2. **Unused variable `concurrentKey` (server.js:1060)** — `const concurrentKey = \`generate_concurrent:${auth.user.id}\`` is assigned but never used. The actual tracking uses `_activeGenerations.get(auth.user.id)` directly. Dead code — remove it.

3. **Test comment mismatch in quota test (test line 181)** — Comment says "do NOT decrement quota on failure — user gets to retry" but then the test simulates decrement of the active count (not quota) without clarifying the distinction. This is not a bug but creates confusion. The comment refers to quota credit, but the code decrement is for the concurrent counter — two separate things.

4. **image_count clamping edge case in tests (test line 31)** — Test asserts `clamp(0) === 6` with the comment "0 is falsy, defaults to 6". This is correct for the `parseInt(v) || 6` pattern, but it means a user who explicitly requests 0 images cannot get 0 images. This is an intentional design decision, but it goes undocumented and untested at the API layer in server.js (where `parseInt(rawImageCount) || 6` applies the same falsy coercion to `0`).

## Patterns Established

1. **Deferred quota decrement pattern** — quota is checked eagerly (before queuing) but consumed (logUsage) only after confirmed job completion via jobQueue.subscribe callback. Active-concurrent count tracks job slots separately from quota credits.

2. **Resolved keys in job config** — API keys resolved at HTTP request time via `keyManager.resolveAllKeys()` and stored in `config.resolvedKeys` on the job. The subprocess receives them via environment variable injection inside `_executeGenerate`. CLAUDECODE is stripped to prevent nested CLI session errors.

3. **CHAINIQ_ env var namespace** — All generation parameters are injected as `CHAINIQ_TOPIC`, `CHAINIQ_LANGUAGE`, `CHAINIQ_FRAMEWORK`, `CHAINIQ_CSS_FRAMEWORK`, `CHAINIQ_IMAGE_STYLE`, `CHAINIQ_IMAGE_COUNT`, `CHAINIQ_DOMAIN_HINT`, `CHAINIQ_PROJECT_DIR` into the subprocess environment.

4. **Structured SSE event shape** — progress events carry `{event, step, total, phase, percent}`. Complete events carry `{event, article_id, file_path, word_count}`. Error events carry `{event, message, step}`. SSE stream uses named events (`event: progress\n`, `event: complete\n`, `event: error\n`) for EventSource filtering.

5. **4-layer settings merge priority** — Request body > auto-config detection > user settings > hardcoded defaults. Each layer validates against an allowlist before accepting the value. Invalid values fall through to the next layer silently.

6. **JobQueue autoProcess flag** — `autoProcess: true` in production triggers `setImmediate(() => this._processNext(token))`. Tests can set `autoProcess: false` to enqueue without immediately processing, enabling assertion on queued state.

7. **Output size guard on subprocesses** — 4MB combined stdout+stderr limit. If exceeded, subprocess is SIGTERM'd and job fails with `err.step` set to the last known stage for targeted error reporting.

## Overall Notes

The implementation is architecturally sound. All major acceptance criteria are met except for one functional bug (AC-7: logUsage call-site mismatch) and one test coverage gap (AC-8: no integration test for the trigger response shape and no test asserting quota credit is preserved on failure).

The logUsage bug is the blocking failure. It causes usage tracking to silently record `user_id: undefined` on every successful generation, breaking quota accounting at the database level. The fix is a one-line change.

The concurrent generation tracking (AC-9) is cleanly implemented with the `_activeGenerations` in-memory Map and correctly decrements on both success and failure paths (the quota credit and the concurrent slot are rightly treated as separate concerns).

The SSE implementation (AC-6) is the strongest part: named event types, structured payloads with step/phase metadata, fallback for unknown stages, proper `req.on('close', unsub)` cleanup, UUID validation on the job ID, and query-param token support for EventSource clients that cannot set headers.

**Required fixes before PASS:**
1. Fix `logUsage` call at server.js:1169 — use positional args `(auth.token, auth.user.id, 'generate', topic)`.
2. Remove dead `concurrentKey` variable at server.js:1060.
3. Add at least one integration test asserting the trigger response shape (`job_id`, `status: 'queued'`, `progress_url`).

## Observations & Risks (Non-Blocking)

### 1. Trigger idempotency gap (minor risk)
Neither `CREATE TRIGGER update_user_settings_updated_at` (in 017) nor `CREATE TRIGGER update_api_keys_updated_at` (in 018) uses `OR REPLACE` or `IF NOT EXISTS`. PostgreSQL 14+ added `CREATE OR REPLACE TRIGGER` but not `IF NOT EXISTS` for triggers. If these migrations are re-run against a live database where the triggers already exist, they will throw an error. The criterion only specifies tables and columns, which are fully idempotent — but this is a real operational risk for anyone re-running migrations.

**Mitigation suggestion for a future task:** Wrap trigger creation in a `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL; END $$;` block, or use `DROP TRIGGER IF EXISTS` before `CREATE TRIGGER`.

### 2. edits_per_day column default vs. plan table (cosmetic inconsistency)
The SQL column default is `edits_per_day INTEGER DEFAULT 10`, but the plan defaults table in the comments shows free = 5. This is not a bug — column defaults are application-level fallbacks, not plan enforcement (which happens in application logic). The discrepancy is cosmetic but could confuse future developers reading the schema cold.

### 3. Test quality
The test file in `tests/migrations.test.js` uses `node:test` with zero npm dependencies — consistent with project conventions. Tests are purely static (regex on SQL file contents), which is appropriate here since there is no local Supabase instance to run against. Coverage is thorough: every column, every RLS policy, every quota column, and the append-only integrity of supabase-setup.sql are all validated. The append-order test (origIdx < newIdx) is particularly well-designed.

## Patterns Established

This is the first task for this project. The following conventions are now established:

1. **Migration naming:** Sequential numeric prefix + descriptive slug: `NNN-description.sql`
2. **Migration headers:** Block comment with migration number, title, phase reference, and purpose description
3. **Idempotent tables:** `CREATE TABLE IF NOT EXISTS public.<name>` for all new tables
4. **Idempotent columns:** `ADD COLUMN IF NOT EXISTS` for all ALTER TABLE additions
5. **Idempotent indexes:** `CREATE INDEX IF NOT EXISTS` for all index creation
6. **RLS pattern — own-row access:** Three separate policies (SELECT via USING, UPDATE via USING, INSERT via WITH CHECK) all using `auth.uid() = user_id`
7. **RLS pattern — role-based access:** `EXISTS (SELECT 1 FROM public.subscriptions WHERE user_id = auth.uid() AND role = '<role>')` for elevated access checks
8. **Trigger pattern:** Shared `update_updated_at_column()` function defined once in 017, reused by 018
9. **Documentation format:** Plan defaults documented as ASCII tables in SQL comments within migration files
10. **supabase-setup.sql protocol:** Append-only — new phases added after a phase header comment block, never modifying existing content
11. **Test pattern:** `node:test` with zero dependencies; static regex validation of SQL content; file existence checks before content checks
12. **Encryption convention:** Encrypted columns use `_encrypted` suffix (e.g., `key_value_encrypted`); display-safe hints use `_hint` suffix
13. **Unlimited value convention:** `-1` represents unlimited for INTEGER quota columns; documented explicitly in migration comments

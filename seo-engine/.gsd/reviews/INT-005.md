## Slop Check

No slop found. Specific checks:

- **No dead code:** Every function is reachable from `generate()` or a builder called by it.
- **No commented-out logic:** Only one explanatory comment (`// Replace with your data source`) which is intentional user guidance.
- **No hardcoded framework assumptions in wrong places:** CSS framework and TS/JS detection gates every conditional output.
- **No raw `<img>` leakage:** `convertContentToJSX` regex transformation is applied to both fallback content and section content paths.
- **No 'use client' pollution:** The page generator never concatenates the overlay content into the page — they are separate file entries.
- **`escapeJSX` correctness:** Escapes `{` and `}` to `{'{'}`/`{'}'}` — covers JSX injection from untrusted title/content strings.
- **External links correctly excluded from Link conversion:** Regex anchors on `href="/"` (forward slash prefix), leaving `https://` untouched.
- **`slugify` has a length cap (80 chars):** Prevents excessively long filesystem paths.

## Patterns Established

1. **Adapter output contract:** `generate(ir)` returns `{ files: [{ path, content }] }`. Path is always relative, starting `app/articles/{slug}/`. Callers write files themselves.
2. **IR shape:** `projectConfig.cssFramework === 'tailwind'` and `projectConfig.uiLibrary === 'shadcn' || projectConfig.hasShadcn === true` are the detection flags for CSS and UI library branching.
3. **Server/client split pattern:** Main page = zero React hooks, zero `'use client'`. Interactive widgets = separate file, `'use client'` first line.
4. **Content-to-JSX transform approach:** HTML string transformation via regex (class→className, for→htmlFor, img→Image, internal a→Link). Appropriate for code-generation context; not a runtime DOM parser.
5. **CSS Module class naming:** All module keys use camelCase (`.container`, `.heroImage`, `.codeBlock`, `.tocLink`). CSS uses logical properties (`margin-inline`, `padding-inline-start`) for RTL compatibility.
6. **FAQ as JSON-LD:** Embedded via `dangerouslySetInnerHTML` on `<script type="application/ld+json">`. Schema.org FAQPage type with Question/Answer structure.
7. **`node:test` + `assert/strict`:** Zero-dependency test pattern. All fixtures defined as plain objects at the top of the file. One describe block per concern.

## Minor Observations (Non-blocking)

1. `generateStaticParams` returns a hardcoded single-item array (`[{ slug: '...' }]`). This is correct as a scaffold — the comment instructs developers to replace it. No action needed.
2. The `layout.tsx` generator does not export a `metadata` object. This is architecturally correct (page-level metadata supersedes layout metadata in Next.js App Router). No action needed.
3. `convertContentToJSX` does not handle self-closing void tags like `<br>` or `<input>` — they would render as invalid JSX. This is acceptable scope for a code-generation adapter focused on article content; a future hardening pass could add void-tag self-closing.
4. The edit overlay uses `process.env.NEXT_PUBLIC_BRIDGE_URL` but does not yet wire click-on-section editing to the bridge server. The bridge URL is defined in the component but no fetch call is made. Acceptable for the current sprint scope; the overlay is a UI scaffold.

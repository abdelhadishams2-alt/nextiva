# Audit: Performance & Scalability

> **App:** article-engine (ChainIQ)
> **Dimension:** E1-C
> **Date:** 2026-03-26
> **Rounds completed:** 3/3

---

## Score: 4/10 — Needs Work

The system is correctly scoped as a single-user local tool, and within that constraint it functions. However: no caching on the hot auth path, synchronous filesystem I/O on the event loop, unbounded memory growth in the edit subprocess, no queue for competing requests, no image optimization pipeline, and no resilience pattern when Supabase is unavailable.

---

## Round 1 Findings (Surface Scan)

### Generated HTML File Size
No client-side bundle. Output is a standalone `.html` file per article. Assembles 8-12+ structural components (from 193 blueprints), 4-6 inline images, two-column layout shell, sticky sidebar TOC, edit UI controls, and all inline JavaScript. No minification. Estimated 150-400 KB raw HTML per article before images. No size budget enforced.

### Image Optimization
Images generated by Gemini MCP as PNG format only (`article-{slug}-{N}.png`). No WebP/AVIF conversion. No `loading="lazy"` attribute. No `srcset` or responsive sizing. 4-6 unoptimized PNGs per article.

### N+1 Patterns
Every `/apply-edit` and `/auth/verify` triggers two sequential Supabase HTTP requests: `verifyToken` (GET `/auth/v1/user`) then `getSubscription` (GET `/rest/v1/subscriptions?user_id=eq.{id}`). Sequential awaits, not concurrent. Classic N+1-style: one logical operation = two blocking network round-trips.

### Caching
Zero caching. No in-process cache for token verification or subscription lookups. `loadConfig()` has a file-mtime check (disk micro-optimization) — the only caching in the entire system. Every request re-verifies against Supabase's live API.

### Concurrent Edit Handling
Single in-process `activeEdit` boolean flag. First `/apply-edit` request locks; all subsequent receive `409 Conflict`. Hard single-tenant constraint: only one edit at any time, server-wide.

## Round 2 Findings (Deep Dive)

### `/apply-edit` Under Load
- `spawn('claude', ['-p'])` with 600,000ms (10 min) timeout
- stdout/stderr accumulated as unbounded in-memory strings — no size limit
- `activeEdit` mutex is module-level memory, not distributed. Server crash loses the flag but Claude subprocess continues running
- No queue. No `Retry-After` header on 409. No notification mechanism.
- **10 concurrent requests:** One wins, nine get 409 immediately with no retry guidance

### Supabase Query Efficiency
- `subscriptions?user_id=eq.{userId}` — relies on index; no enforcement from codebase
- `usage_logs` composite query needs `(user_id, action, created_at)` index — grows linearly without it
- `adminListUsers()` fetches ALL users AND ALL subscriptions without pagination — breaks at scale

### Synchronous Blocking Operations
- `fs.existsSync()` on every `/apply-edit` — blocks event loop
- `fs.writeFileSync()` on every login — blocks event loop
- `fs.statSync` and `fs.readFileSync` in `loadConfig()` — blocks on every request path

### Memory Leak Risks
- `rateLimitMap` cleanup only fires at >100 entries. Below that, entries are never evicted
- stdout/stderr strings grow without bound during 10-min Claude subprocess
- No `server.keepAliveTimeout` or `server.headersTimeout` — clients can hold connections indefinitely

### Dependency Failure Modes
- **Supabase down:** Returns misleading 401 ("Invalid or expired token") instead of 503. No circuit breaker, no retry, no fallback. Server becomes non-functional for edits.
- **Gemini MCP down:** Graceful degradation to Level 1 fallback (placeholder HTML + exported prompts). Well-designed.
- **Claude CLI not found:** Correctly handled — 500 with clear error message.

## Round 3 Recommendations

### Lowest-Effort, Highest-Impact Improvement
**Cache token verification results with 30-second TTL.** Eliminates 2 HTTP round-trips per request. ~15 lines of code. Impact: auth latency drops from ~200-600ms to near-zero for cached tokens.

### What Must Change for 100x Load
1. `activeEdit` must become per-user/per-article queue (not global mutex)
2. Shared coordination layer (Redis/DB lock) for horizontal scaling
3. Auth token caching mandatory
4. Replace sync `fs.*` with `fs.promises.*`
5. `adminListUsers()` must be paginated
6. Add `server.keepAliveTimeout` and `server.headersTimeout`

### Quick Wins
| Item | Effort | Impact |
|------|--------|--------|
| Token verification cache (30s TTL) | 1 hour | High |
| Parallelize `verifyToken` + `getSubscription` with `Promise.all` | 30 min | Medium |
| Replace sync fs calls with async | 30 min | Low-medium |
| Add `Retry-After` header to 409 | 15 min | UX |
| Add `loading="lazy"` to generated images | 1 hour | High |
| Convert PNG to WebP in image pipeline | 2-3 hours | High |
| Enforce stdout size limit (4MB) in `/apply-edit` | 1 hour | Medium |
| Periodic `rateLimitMap` eviction (setInterval 60s) | 30 min | Low |

---

## P0 Issues (Fix Immediately)
- Unbounded stdout/stderr memory growth during 10-min Claude subprocess
- Misleading 401 when Supabase is down (should be 503)

## P1 Issues (Fix Before Scaling)
- No caching on auth path (2 remote HTTP calls per request)
- Synchronous filesystem I/O on event loop
- Global `activeEdit` mutex prevents any concurrent editing
- No image optimization (PNG only, no lazy loading)
- `adminListUsers()` not paginated

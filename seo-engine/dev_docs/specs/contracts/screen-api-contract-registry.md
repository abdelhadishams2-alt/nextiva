## Response Envelope Pattern

### Success Response (single entity)

```json
{
  "success": true,
  "data": { "id": "uuid", "...": "..." }
}
```

### Success Response (list/paginated)

```json
{
  "success": true,
  "data": [ "..." ],
  "meta": { "total": 100, "page": 1, "limit": 20, "totalPages": 5 }
}
```

### Error Response

```json
{
  "error": "Unauthorized",
  "message": "Token expired"
}
```

### SSE Stream Response

```
Content-Type: text/event-stream

data: {"stage":"research","progress":0.3,"message":"Researching topic..."}
data: {"stage":"complete","progress":1.0,"result":{...}}
```

**Frontend rule:** All API calls must unwrap `.data` before passing to components.

## Authentication & Authorization

### Auth Flow

1. User submits credentials ‚Üí `POST /auth/login` ‚Üí receives JWT
2. JWT stored in httpOnly cookie (dashboard) or Authorization header (API)
3. Every authenticated request: Bearer token validated via `verifyAuth()` middleware
4. Auth cache: SHA-256 hash of token ‚Üí cached result (30s TTL)
5. Admin endpoints: require `is_admin` flag in user metadata

### Role Matrix

| Endpoint Pattern | Admin | Active User | Pending User | Anonymous |
|------------------|-------|-------------|--------------|-----------|
| `POST /auth/*` | -- | -- | -- | Yes |
| `GET /health` | -- | -- | -- | Yes |
| `GET /api/*` (list/detail) | Yes | Yes (own data) | No | No |
| `POST /api/*` (create/action) | Yes | Yes (quota enforced) | No | No |
| `PUT /api/*` (update) | Yes | Yes (own data) | No | No |
| `DELETE /api/*` | Yes | No (except own API keys) | No | No |
| `/admin/*` | Yes | No | No | No |
| `GET /api/admin/*` | Yes | No | No | No |

## Consistency Notes & Discrepancies

### 1. Performance Screen vs Feedback Loop Service ‚Äî Path Naming

**Issue:** The Performance screen spec (Screen 14) uses `/api/performance/*` paths, but the Feedback Loop service spec uses `/api/feedback/*` paths.

**Resolution:** Use **`/api/performance/*`** for the 4 user-facing dashboard endpoints (summary, chart, articles, article detail) and **`/api/feedback/*`** for the 5 backend/admin endpoints (accuracy, portfolio, recalibrate, history, rollback). This keeps the user-facing API clean (`/performance` matches the `/performance` route) while the admin/backend API uses the service name. Both are served by the same Feedback Loop service.

### 2. Voice Profiles Screen ‚Äî Endpoint Path Variations

**Issue:** Screen spec uses `PUT /api/voice/personas/:id/default` and `GET /api/voice/analysis/:sessionId`, while service spec uses `POST /api/voice/personas/:id/set-default` and `GET /api/voice/analyze/progress/:sessionId`.

**Resolution:** Adopt the **service spec** paths as canonical (POST for set-default is more RESTful for actions; `/analyze/progress/` is consistent with other SSE patterns). Update screen spec references during implementation.

### 3. Admin API V2 Endpoints

**Issue:** Screen 5 (User Management) references `/api/users/*` V2 endpoints that don't exist yet. Current implementation uses legacy `/admin/*` endpoints.

**Resolution:** V2 endpoints are planned for Phase 5+ implementation. Legacy endpoints serve as fallback. Both will coexist during transition.
